3. CRT Absolute Positioning

The turret uses a Chinese Remainder Theorem (CRT) technique with two CANcoders geared at different ratios to resolve absolute turret position across the full 334° range (beyond a single encoder's 360° ambiguity).

CRT Position Calculation Flow
Turret Initialization / Reset
Read CANcoder 1 (absolute)
Ratio: 90/13 teeth = 6.923:1
CAN ID 41
Read CANcoder 2 (absolute)
Ratio: 90/14 teeth = 6.429:1
CAN ID 42
diff = enc1 − enc2
if (diff < 0) diff += 1.0
position = diff × (1 / (6.923 − 6.429)) × 360°
Seed SparkMax relative encoder with CRT position
m_turretMotor.getEncoder().setPosition(crtPosition)
Runtime: use SparkMax relative encoder
(faster reads, no CAN latency)
Why CRT? Each encoder goes through ~6.9 or ~6.4 full rotations for one turret revolution. Their difference in phase uniquely identifies the absolute turret position without needing a single-speed absolute encoder that covers the full range. The effective factor is 1/(R1 − R2) = 1/(90/13 − 90/14) = 1/(90·(14−13)/(13·14)) = 13·14/90 ≈ 2.022, so position = diff × 2.022 × 360°.

Runtime Position Tracking

After initialization, position comes from the SparkMax's built-in relative encoder with a conversion factor of 360 / 180 = 2 degrees per motor rotation. A drift monitor logs calculateTurretPosition() − getTurretPosition() to SmartDashboard for diagnostics.

Feedforward (Currently Disabled)

A SimpleMotorFeedforward is instantiated with kS=0.02, kV=0.10 but is currently commented out in fastPeriodic(). When active, it would add a velocity-proportional term based on the rate of change of the target angle, helping the turret track moving targets more smoothly.

findBestAngle Logic

The turret has a physical range of [0°, 334°]. Since the robot-relative angle can be any value, findBestAngle() generates three candidate angles (angle−360, angle, angle+360) and selects the in-bounds candidate closest to the current position, minimizing travel distance and preventing wrap-around collisions.

Trench Safety Interlock

When the robot enters a trench region (low ceiling), the hood angle is automatically forced to 0 to prevent mechanical collision. This safety check is built directly into ShooterSubsystem.fastPeriodic() via a Supplier<Pose2d> passed at construction time, which avoids subsystem-requirement conflicts with ShooterCommand:

// In ShooterSubsystem.fastPeriodic() — runs at 100 Hz
if (FieldConstants.kTrenchesRegion.isInRegion(m_poseSupplier.get())) {
    setHoodAngle(ShooterConstants.kHoodAngleMin);
}
// ... then hood PID runs with the (possibly overridden) setpoint
Note: The previous implementation used a Trigger/InstantCommand in RobotContainer that required m_shooter, which conflicted with ShooterCommand also requiring the same subsystem. Moving the check into the subsystem's own fast loop eliminates this conflict and also runs at 100 Hz instead of 50 Hz.

8. Shoot-on-the-Move (SOTM)

LaunchCalc.findTargetOnTheMove() implements an iterative algorithm to compute a corrected aim point that accounts for the robot's velocity and the ball's time of flight. It is currently disabled in RobotContainer.fastPeriodic() (commented out).

SOTM Iterative Algorithm
Inputs: robotPose, targetPose, robotVelocity
iterations = 5
Initialize: newTarget = targetPose
For each iteration:
1. distance = |robotPose − newTarget|
2. timeOfFlight = TimeLUT.get(distance)
3. newTarget = target − velocity × ToF × 1.2
4. If moving: offset target slightly away from robot (×0.1)
Return corrected newTarget

11. Timing & Execution Rates

Execution Timeline — One Robot Cycle
Every 10 ms (100 Hz) — fastPeriodic
100 Hz

① DriveSubsystem.fastPeriodic() — set swerve module states
② TurretSubsystem.fastPeriodic() — PID → motor output
③ ShooterSubsystem.fastPeriodic() — flywheel PID+FF, hood PID
④ FindTarget.getTarget() → update currentTarget
Every 20 ms (50 Hz) — CommandScheduler
50 Hz

① AutoAimTurret.execute() — calculateSetpoint()
② ShooterCommand.execute() — LaunchCalc lookup + feed check
③ DriveSubsystem.periodic() — odometry update, vision fusion
④ TurretSubsystem.periodic() — SmartDashboard telemetry
⑤ ShooterSubsystem.periodic() — SmartDashboard telemetry
Design rationale: The PID control loops (turret motor, flywheel, hood) run at 100 Hz for tighter control and reduced latency. The setpoint calculations and command logic run at 50 Hz since they depend on pose estimation which updates at a similar rate. This separation ensures responsive motor control while keeping CPU usage reasonable.

12. Recommended Improvements

The following improvements were identified from a code review of the turret targeting pipeline. They are ordered by estimated impact — bugs first, then performance, then quality-of-life.

🐛 Bug Fixes

1. ✅ Telemetry Copy-Paste Bug — Encoder 2 reads Encoder 1 FIXED

In TurretSubsystem.periodic(), the "Encoder 2 Relative Angle" telemetry was reading m_encoder1 instead of m_encoder2. This has been fixed — the line now correctly reads m_encoder2.

// FIXED: now correctly reads m_encoder2
SmartDashboard.putNumber("Turret/Encoder 2 Relative Angle",
    m_encoder2.getPosition().getValueAsDouble() * 360 / kEncoder2Ratio);

2. ✅ Trench Hood Override Conflicts with ShooterCommand FIXED

The trench safety trigger previously created an InstantCommand requiring m_shooter, which conflicted with ShooterCommand also requiring the same subsystem. This has been fixed — the trench check is now built into ShooterSubsystem.fastPeriodic() using a Supplier<Pose2d> passed at construction time. The conflicting Trigger has been removed from RobotContainer.

// ShooterSubsystem now takes a pose supplier in its constructor:
new ShooterSubsystem(m_robotDrive::getPose)

// In fastPeriodic(), trench safety runs at 100 Hz with no subsystem conflicts:
if (FieldConstants.kTrenchesRegion.isInRegion(m_poseSupplier.get())) {
    setHoodAngle(ShooterConstants.kHoodAngleMin);
}

3. ShooterCommand.initialize() Missing @Override

ShooterCommand declares public void initialize(), execute(), end(), and isFinished() without @Override annotations. If the method signature doesn't exactly match the parent, it silently becomes a new method instead of overriding. This means the flywheel reset in initialize() might not actually run. Always use @Override.

🔥 High Impact Improvements

4. Enable Turret Feedforward for Tracking

The feedforward is fully computed (relativeAngularVelocity is calculated every cycle) but commented out in fastPeriodic(). For a turret that needs to track a moving target (especially while the robot rotates), feedforward is critical to reduce tracking lag. Without it, the PID must react to error after it appears, causing the turret to always trail the target.

Current (PID only):
Target moves → error builds → PID reacts → turret lags
With feedforward:
Target moves → FF predicts motion → PID corrects residual → near-zero lag 
// In TurretSubsystem.fastPeriodic(), uncomment and add:
double output = m_turretPID.calculate(getTurretPosition())
    + Math.signum(m_turretPID.getError())
    * Math.abs(m_turretFeedforward.calculate(relativeAngularVelocity));

5. ✅ Gate Shooting on Turret Alignment — Not Just Flywheel Speed IMPLEMENTED

This has been implemented. ShooterCommand and ManualShooterCommand now accept a TurretSubsystem parameter (read-only, not added to requirements). Ball feeding is gated on all three conditions: flywheel speed, hood angle, and turret alignment. A new isHoodReady() method was also added to ShooterSubsystem.

Before: Feed when flywheel ready only
Now: Feed when flywheel ready AND hood ready AND turret.atSetpoint() 
// ShooterCommand.execute() — implemented triple gate:
if (m_shooterSubsystem.isShooterReady()
    && m_shooterSubsystem.isHoodReady()
    && m_turretSubsystem.atSetpoint()) {
    m_shooterSubsystem.setSpindexer(true);
    m_shooterSubsystem.setTransfer(true);
}

// isHoodReady() in ShooterSubsystem:
public boolean isHoodReady() {
    return Math.abs(m_hoodAnglePID.getSetpoint() - getHoodAngle()) < 0.03;
}

6. Enable Shoot-on-the-Move (SOTM)

The iterative SOTM algorithm in LaunchCalc.findTargetOnTheMove() is fully implemented but commented out in RobotContainer.fastPeriodic(). Enabling it would let the robot score while translating, significantly improving cycle times. The algorithm already handles velocity compensation and time-of-flight correction.

// In fastPeriodic(), replace:
currentTarget = FindTarget.getTarget(m_robotDrive.getPose());

// With:
ChassisSpeeds fieldSpeeds = ChassisSpeeds.fromRobotRelativeSpeeds(
    m_robotDrive.getRobotRelativeSpeeds(), m_robotDrive.getPose().getRotation());
currentTarget = LaunchCalc.findTargetOnTheMove(
    m_robotDrive.getPose(),
    FindTarget.getTarget(m_robotDrive.getPose()),
    new Translation2d(fieldSpeeds.vxMetersPerSecond, fieldSpeeds.vyMetersPerSecond));

7. Reset PID Integral on Large Setpoint Changes

When the turret setpoint changes dramatically (e.g., switching from Hub to wall target, or cardinal direction buttons), the accumulated integral from the old setpoint is invalid and can cause significant overshoot at the new setpoint. Fix: Reset the PID when the setpoint changes by more than a threshold:

public void setSetpoint(double angle) {
    double clampedAngle = MathUtil.clamp(angle, 0, kTurretMaxRotation);
    if (Math.abs(clampedAngle - m_turretPID.getSetpoint()) > 10.0) {
        m_turretPID.reset();  // clear accumulated integral
    }
    m_turretPID.setSetpoint(clampedAngle);
}
⚡ Medium Impact Improvements

8. Add Flywheel Pre-Spin

ShooterCommand.initialize() calls reset() which sets the flywheel to 0 RPM. Every time the driver pulls the trigger, the flywheel must spin up from zero, wasting ~0.5–1s. Instead, keep the flywheel spinning at a baseline speed whenever auto-aim is active, or start the flywheel as soon as the robot enters its alliance zone:

// Option A: idle flywheel speed as a default command
m_shooter.setDefaultCommand(new RunCommand(() -> {
    if (FindTarget.shouldScoreHub(m_robotDrive.getPose())) {
        m_shooter.setShooterParameters(3100, ShooterConstants.kHoodAngleMin);
    }
}, m_shooter));

9. Add Software Soft Limits to Turret Motor

There are no software limits on the SparkMax. If the PID overshoots or the encoder drifts, the turret could slam into a mechanical hard stop. Add soft limits in the motor config:

motorConfig.softLimit
    .forwardSoftLimit(kTurretMaxRotation)
    .forwardSoftLimitEnabled(true)
    .reverseSoftLimit(0)
    .reverseSoftLimitEnabled(true);

10. Periodic CRT Drift Correction

The drift between the CRT absolute position and the SparkMax relative encoder is logged ("Turret/Drift value") but never corrected. Over a long match, the relative encoder can drift several degrees. Fix: Periodically re-seed the relative encoder when the turret is stationary and the drift exceeds a threshold:

// In periodic(), add:
double drift = calculateTurretPosition() - getTurretPosition();
if (Math.abs(drift) > 2.0 && Math.abs(m_turretMotor.getEncoder().getVelocity()) < 0.1) {
    m_turretMotor.getEncoder().setPosition(calculateTurretPosition());
}

11. Clamp Hood Angle Setpoint

setHoodAngle() passes the value directly to the PID with no bounds checking. If the LaunchCalc LUT extrapolates beyond the table or a bad value is passed, the hood could be driven past its limits. Add a clamp:

public void setHoodAngle(double angle) {
    m_hoodAnglePID.setSetpoint(
        MathUtil.clamp(angle, ShooterConstants.kHoodAngleMin, ShooterConstants.kHoodAngleMax));
}

12. Add D-term to Turret PID

kTurretD is 0.0. For a turret tracking a moving target, a small derivative term would dampen oscillations around the setpoint and reduce overshoot during fast slews. Start with kD = 0.001 and tune from there. The fast 100 Hz loop rate makes D-term noise less of a concern.

✨ Quality-of-Life Improvements

13. Driver Feedback — Haptic/LED When Ready to Fire

There is no feedback to the driver indicating when the turret + flywheel + hood are all locked on target. The driver is shooting blind. Add controller rumble when the system is ready:

// In RobotContainer.periodic():
boolean readyToFire = m_turret.atSetpoint() && m_shooter.isShooterReady();
m_driverController.setRumble(RumbleType.kBothRumble, readyToFire ? 0.3 : 0);

14. Simplify calculateSetpoint Coordinate Transform

The current angle conversion uses a confusing double-inversion:

findBestAngle(kTurretFrontAngle - (360 - robotRelativeAngle), (360 - currentPosition))

This is difficult to reason about and debug. Consider refactoring to explicit, well-named intermediate variables:

// Robot-relative angle from atan2
double fieldRelativeAngle = robotRelativeAngle + robotPose.getRotation().getDegrees();
// Convert to turret coordinate system (turret 0° = some physical reference)
double turretAngle = kTurretFrontAngle - (360 - robotRelativeAngle);
double targetAngle = findBestAngle(turretAngle, getTurretPosition());  // use actual position, not inverted

15. Move Auto-Aim Toggle to a Subsystem State

The autoAimTurret boolean lives as a raw field in RobotContainer and is toggled by button bindings that don't require any subsystem. This means the turret has no default behavior when auto-aim is off — it just holds whatever setpoint was last written. Consider making this a proper state in TurretSubsystem with a default command that always runs (either tracking or holding a manual setpoint from operator joystick input).
